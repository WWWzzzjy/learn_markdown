10. **L1和L2正则化分别是什么，什么场景适合使用呢？**

    L1 和 L2 正则化，本质上都是在原始损失函数后面加一个“参数惩罚项”，目的是限制模型复杂度，防止过拟合。

    L1 正则化是在损失函数后面加上**参数绝对值之和**，抑制过拟合，让参数变得稀疏；

    L2 正则化是在损失函数后面加上**参数平方和**，倾向于让参数变小，让模型参数整体更平滑，防止某些权重过大，提升泛化能力

14. 在训练一个百或千亿参数级别的 LLM 时，你会面临哪些主要的工程和算法挑战？（例如：显存、通信、训练不稳定性等）

    显存与存储压力：因为训练一个超大模型时，GPU 上放参数、梯度、优化器状态、激活值

    解决方法:

    **混合精度训练**，比如 BF16 / FP16

    **ZeRO / FSDP**，把参数、梯度、优化器状态做分片

    **CPU / NVMe Offload**，把部分状态卸载到更慢的存储层

    **FlashAttention**，减少 attention 的显存访问和中间张量

    

    分布式并行与通信开销：当模型大到单卡甚至单机都放不下时，就必须做分布式训练，通讯会成为瓶颈

    并行策略：DP、TP、PP、SP、Expert Parallel（MoE）

    数据并行要做梯度 All-Reduce

    张量并行层间要频繁做 All-Gather / Reduce-Scatter

    Pipeline Parallel 会有 stage 间传输和 bubble 问题

    MoE 还会带来 token 到 expert 的 All-to-All 通信

    

    训练稳定性问题：模型一旦进入百亿、千亿级别，训练不稳定会被明显放大。

    梯度爆炸 / 梯度异常、loss spike、数值溢出、训练后期发散、小 bug 在长时间训练中被放大

    

    数据质量与训练效率挑战：因为如果数据质量差、重复多、噪声重，就会出现训练效率低、泛化性差

    

    算法上的相关问题：Scaling law 下的最优配比：参数量、数据量、训练 token 数之间要平衡，不是参数越大越好

# RLHF

1. 和传统SFT相比，RLHF旨在解决语言模型中的哪些核心问题？为什么说SFT本身不足以实现我们期望的“对齐”目标？

   和传统 SFT 相比，RLHF 主要是为了解决“人类偏好无法通过单一标准答案充分表达”的问题。
    也就是说，SFT 能让模型学会“像样地回答”，但很难进一步保证模型回答 更符合人类偏好、更有帮助、更安全、更自然（3H）。而 RLHF 的目标，就是在 SFT 的基础上，继续把模型往“人类真正喜欢的回答方式”上推近一步。

1. 请详细阐述经典RLHF流程的三个核心阶段。在每个阶段，输入是什么，输出是什么，以及该阶段的关键目标是什么？

   阶段1：SFT

   - **输入**：prompt + 人类示范回答
   - **输出**：SFT 模型
   - **目标**：让模型具备指令跟随和基本对话能力

   阶段2：Reward Model

   - **输入**：prompt + 多个候选回答 + 人类偏好排序
   - **输出**：奖励模型 RM（BT）
   - **目标**：学习一个可以近似人类偏好的奖励函数

   阶段3：RL（PPO）

   - **输入**：prompt + SFT 初始化策略 + RM + reference model
   - **输出**：RLHF 后的对齐模型
   - **目标**：在不偏离原始语言能力太多的前提下，最大化人类偏好奖励

   

2. 在RM训练阶段，我们通常收集的是成对比较数据，而不是让人类标注者直接给回复打一个绝对分数。你认为这样做的主要优势和潜在的劣势分别是什么？

   成对比较的核心优势：更符合人类自然偏好表达方式，标注更稳定（相对判断的标注一致性高）、排序目标更清晰；

   核心劣势：只能提供相对顺序信息，缺少绝对质量刻度，而且偏好数据本身可能有噪声、偏差和不传递性。

6. 什么是“奖励作弊/奖励黑客”（Reward Hacking）？请结合一个具体的LLM应用场景给出一个例子，并探讨几种可能的缓解策略。

   模型并没有真正学会我们想要的目标，而是学会了利用奖励函数、奖励模型或者评价机制中的漏洞，用一种“投机取巧”的方式拿到高分。

   模型可能学会一些“骗过奖励模型”的模式，而不是学到真正更好的行为。比如：

   - 回答变得过长
   - 过度迎合
   - 套模板化表述
   - 表面上更像高分答案，但实际质量下降

7. RLHF流程复杂且不稳定。近年来出现了一些替代方案，例如DPO。请解释DPO的核心思想，并比较它与传统RLHF（基于PPO）的主要区别和优势。

   DPO不使用奖励模型，拿偏好数据直接做带 reference 约束的对数概率对比学习。

   对于一条数据 $(x, y_w, y_l)$，DPO 会比较：

   - 当前策略对 winner 和 loser 的对数概率差
   - 参考模型对 winner 和 loser 的对数概率差

   如果当前策略相对于参考模型，更倾向于 winner 而不是 loser，那么 loss 就会更小；反之就会被更新。

9. 你知道Deepseek的GRPO吗，**它和PPO的主要区别是什么**？优劣是什么？

   GRPO 是 DeepSeek 提出的 PPO 变体，核心思想是取消 PPO 里额外训练的 value/critic 模型，对同一个 prompt 一次采样多条回答，用这组回答的相对奖励经过组均值和组标准差归一化后，直接作为优势信号来更新 policy。相比 PPO，GRPO 最大的区别是不用单独学 value function，因此显存和训练开销更低；同时它更适合数学、代码这类有可验证奖励的推理任务。它的优势是简单、高效、资源占用低，

# Agent

+++

1. 你如何定义一个基于 LLM 的智能体（Agent）？它通常由哪些核心组件构成？

   Agent = LLM（推理核心）+ 感知 + 记忆 + 工具 + 行动 的自主决策系统。它不只是被动地回答问题，而是能感知环境、制定计划、调用工具、持续行动，直到完成复杂目标

   LLM：推理核心。接受感知模块，参考记忆，与工具交互。常用ReAct范式驱动

   感知模块perception：处理多模态的输入（文本、图像、音频、结构化数据包括API返回的JSON和哦工具调用结果）

   记忆模块Memory：短期记忆、长期记忆、情景记忆

   工具调用Tool use：网络搜索、执行 Python 代码、调用外部 API、读写文件、操控浏览器等。LLM 通过生成结构化的 function call 来触发工具

   

2. 请详细解释 ReAct 框架。它是如何将思维链和行动结合起来，以完成复杂任务的？

   Thought → Action → Observation。让 LLM 在每一步都先"思考"再"行动"，并将行动结果作为新的观察融入下一轮推理

   思维链只推理不行动，知识静态、无法验证推理是否正确、容易幻觉。

   标准 CoT 的推理链完全在 LLM 内部，靠参数记忆支撑，一旦超出训练知识就会出错。ReAct 在每个 Thought 之后插入一个 Action→Observation 环节，用真实世界的信息校正推理链，让 Thought 始终有事实依据。

   

3. 在 Agent 的设计中，“规划能力”至关重要。请谈谈目前有哪些主流方法可以赋予 LLM 规划能力？（例如 CoT, ToT, GoT等）

   CoT让模型不直接生成答案，而是通过问题-推理链-答案，展开中间推理过程。有两种触发方式：Few-shot结合zero-shot

   ToT 是对 CoT 的重大扩展：把单条推理链变成一棵**搜索树**，在每个节点都探索多个可能的下一步，并用**评估函数**剪枝。

   ToT 的核心组件有四个：**思维生成器**（产生 k 个候选下一步）、**评估器**（给每个节点打分，可用 LLM 自评或外部启发式）、**搜索算法**（BFS 广度优先或 DFS 深度优先）、**剪枝策略**（丢弃低分节点）。适合解数学题、24点游戏、创意写作等需要回溯的任务，但调用 LLM 次数多，成本高。

   ToT 的思维只能父→子单向流动，GoT 打破了这一限制，允许节点**任意合并、聚合、形成环**，更接近人类真实的思考结构。

   GoT 引入了三类核心操作：**分解**（将大任务拆为并行子任务）、**聚合**（将多路思维合并为一个更好的结论）、**精炼**（对节点自我修正，形成有向环）。适合文档排序、集合运算等可并行化的任务，效率显著高于 ToT。

   

4. Memory是 Agent 的一个关键模块。请问如何为 Agent 设计短期记忆和长期记忆系统？可以借助哪些外部工具或技术？

   短期记忆：对应 Agent **当前会话/任务**中的信息存储，本质上是对 LLM **Context Window** 的管理。

   * 完整历史对话。信息无损，但容易超出上下文窗口限制
   * 滑动窗口。保留最近N轮对话，丢弃更早的历史。长度控制，但是丢失早期关键信息
   * 对话摘要压缩：定期对历史对话压缩成摘要，保留摘要+近N轮对话
   * 结构化状态：维护一个结构化的工作区，记录任务状态、中间变量等

   长期记忆：长期记忆对应 Agent **跨会话、跨任务**的知识积累，必须借助**外部存储**实现持久化。 通过RAG 按需取回

   * 语义记忆——知识库的形式：存储用户手册、产品文档、历史报告、背景资料，
   * 情节记忆——历史经历：存储 Agent 过去的完整对话记录、任务执行轨迹，供将来参考
   * 程序记忆——技能和经验：Agent的操作经验、任务模板、最佳实践。一般固化进 System prompt，通过FT让模型权重学会

   

5. Tool Use是扩展 Agent 能力的有效途径。请解释 LLM 是如何学会调用外部 API 或工具的？（可以从 Function Calling 的角度解释）

   LLM 的本质是文本输入、文本输出。

   tool use本质是让 LLM 输出结构化的"意图描述"，由外部系统解析并真正执行，再把结果还给 LLM

   * SFT：通过专门构造的训练数据学会的。训练数据包含：返回工具调用的格式、根据工具调用结果继续生成
   * RL：通过强化学习奖励"正确调用工具并给出好回答"的行为，惩罚"乱调工具"或"参数填错"的行为，提升工具调用的准确性

   Function的完整流程：

   * 开发者定义工具，用 json scheme 描述工具的名称、功能、参数

   * 工具描述注入prompt

   * 模型决策——调用 / 不调用：输出结构化的 tool call，模型停止生成

   * 外部系统执行工具

   * 结果回注，模型继续生成：工具执行结果作为新的消息追加到对话历史，模型基于结果生成回复

     

6. 请比较一下两个流行的 Agent 开发框架，如 LangChain 和 LlamaIndex。它们的核心应用场景有何不同？

   

7. 在构建一个复杂的 Agent 时，你认为最主要的挑战是什么？

   * 任务的规划与分解的可靠性：这是最根本的挑战。LLM 在短任务上表现不错，但面对需要十几步甚至几十步的复杂任务时，会出现规划跑偏、中间推理步骤出错导致结果完全偏离。这也是为什么 Plan-and-Execute、ReAct等框架被提出——本质上都是在试图让 Agent 的规划更稳定可控。
   * 工具调用的准确性和鲁棒性（工具选择错误、参数幻觉、调用失败处理错误信息、工具数量瓶颈）
   * 记忆和状态管理：Agent 运行过程中需要维护大量状态
   * 安全控制

   构建复杂 Agent 最大的挑战不是单点能力，而是**如何在长链路、真实环境、不可预测输入下保持整体可靠性**；
