3. **为什么要用图，传统的比如关键字检索的方法可行嘛**

传统的方法有哪些：BM25关键字搜索；用RAG的思想进行一个代码向量数据库的构造；使用 Agent 或者 Agentless 方式

传统的方法有哪些不足：

我们在这里把握的一个要点就是：issue 描述的是自然语言，代码库是结构化程序，两者之间存在天然的语义鸿沟。从报错位置到根源，需要跨越多个文件、沿着调用链追溯——这就是所谓多跳推理(multi-hopreasoning)问题。而传统的方法无论是BM25还是RAG用向量相似性的思想，或者说一些agent，都是利用语义相似去检索，但是语义相关不代表依赖相关。真正需要修改的函数可能跟 issue 中描述的内容在语义上毫无关联。

4. **图工具有哪些，各自有什么作用**

将代码库解析为有向异构图，图中包含` directory、file、class、function `四种节点以及定义的四种有向边 `contains、imports、invokes、inherits` ，将代码的**结构依赖关系** **显式编码**为可查询的图结构。

我们为每一个 entity 规定了一种精确的实体ID，组成：实体的文件节点的相对路径：路径~类或者基本函数`.` 类名~方法名

1. `retrieve_entity`：**精确检索实体工具**。输入的是实体精确ID，在图节点中查询，返回节点的 code 属性，也就是代码块，若查找不到直接返回报错信息

2. `traverse_graph`：**图遍历工具**。输入的是实体精确ID，参数可以设定多跳遍历的方向和深度。从一个节点出发，在图上做 BFS，返回当前节点多跳遍历的 **tree** 结构。

3. `search_entity`：**搜寻节点**，输入的是keyword，返回完整实体精确ID。

   内部三级检索：

   * 精准的图节点匹配，返回实体ID
   * 实体名称的倒排索引，返回相关的实体精确ID
   * 稀疏检索：BM25在所有节点的"code"属性中做词频匹配，返回最相关的实体的ID

8. **你提到的并行工具调用是如何做的（量化：检索效率、检索时间、检索质量）**

为了实行并行工具调用并且解决 credit assignment，利用 差分NDCG 和 tool efficiency 作为奖励函数。为了控制轮数限制，在reward中再加一个token长度惩罚项（token/token_budget）

在**轨迹**中，对每轮LLM调用工具返回的实体都记录下来形成一个 entities_history 集合，然后每一次调用都计算当前返回的实体中有多少是在entities history集合中没见过的，计算一个0~1的 tool efficiency。

量化结果：我们分别使用了加上 tool_efficiency 和不加上 efficiency 的奖励函数进行训练，结果证明并行工具调用不仅缩短了平均搜索时间，而且降低了平均搜索轮次 并且NDCG值也有所提高

> 定位错了（NDCG=0）无论多高效奖励都是 0，防止模型学会"快速但乱搜"的策略。

我们主要用到的是：仓库 Repo、这个仓库里修复前的一次 base_commit、问题表述 problem_statement、以及从patch中提取出的 ground-truth。problem statement就是agent的输入，其实就是 github 上对于 repo 的一个 issue ，包括 title 和 description

10. **整一个GRPO的训练流程是怎么样的**
    1. 从 SWE-bench_Lite 数据集加载 Parquet 格式的训练样本。每个 issue 复制 5 份用于 GRPO 组内比较
    2. Rollout：5 份相同的 issue 并行送入LLM，模型通过 vLLM 多轮推理，最终输出包含<trace_locs>
    3. Reward 计算：解析每条轨迹的预测位置，与 ground truth 在文件、函数两个粒度上分别计算 NDCG@5 和 tool_efficiency，轨迹中的多轮工具调用做差分处理。
    4. Advantage 归一化：对同一 issue 的 5 条轨迹做组内归一化，再分配到每条轨迹中每一轮的 reward，再广播到每一个token
    5. 参数更新：用 Clipped Loss 对生成的 token 做梯度更新，同时加 KL 惩罚防止偏离 SFT 初始模型过远

12. **介绍一下VeRL架构**（这个介绍的还行，但是感觉不够深入，要是问的再细一点不一定能回答出来）

VeRL 是一个面向大语言模型后训练的强化学习框架。

在 RLHF 场景里，通常会同时涉及 actor、critic、reward model 等多个模块，而且训练阶段和生成阶段的并行策略、显存占用、通信模式都不一样。

VeRL 最核心的特点是采用了 HybridFlow Controller（混合控制器流） 的架构，融合单控制器（Single-Controller）和多控制器（Multi-Controller）的特点。把强化学习中涉及到的 rollout、reward、policy update、分布式调度、训练推理切换等都进行了抽象。single-controller 用一个中心进程来调度所有计算节点（worker），multi-controller 把所有计算节点在内部都做分布式计算

13. **训练中的Rollout和Training如何去切换（为什么去做、如何去做）** 

原因：因为大模型强化学习里这 训练阶段 和 rollout 阶段的最优并行策略通常并不相同，训练更关注反向传播效率，而 rollout 更关注推理吞吐，所以 VeRL 允许 actor 在两阶段采用不同的 3D 并行配置。

VeRL中使用的是 3D-HybridEngine，让 actor 在 生成阶段 和 训练阶段之间切换时进行参数重分片与资源迁移，目标就是减少这部分显存冗余和通信开销。

训练态权重和生成态权重采用双缓冲管理。训练阶段（training）保留训练态分片，生成态权重可 offload 到 CPU

切换到 rollout 时，再基于训练态的分片通过 all-gather / resharding 组织出生成态权重，并加载到 vLLM 侧执行生成。生成完成后，系统会释放 KV cache，并将权重重新分片回训练态
